8  // This example roughly illustrates the proper way to add multiple players from existing
9  // devices. Notice how InputManager.Devices is not used and no index into it is taken.
10  // Rather a device references are stored in each player and we use InputManager.OnDeviceDetached
11  // to know when one is detached.
12  //
13  // InputManager.Devices should be considered a pool from which devices may be chosen,
14  // not a player list. It could contain non-responsive or unsupported controllers, or there could
15  // be more connected controllers than your game supports, so that isn't a good strategy.
16  //
17  // To detect a joining player, we just check the current active device (which is the last
18  // device to provide input) for a relevant button press, check that it isn't already assigned
19  // to a player, and then create a new player with it.
20  //
21  // NOTE: Due to how Unity handles joysticks, disconnecting a single device will currently cause
22  // all devices to detach, and the remaining ones to reattach. There is no reliable workaround
23  // for this issue. As a result, a disconnecting controller essentially resets this example.
24  // In a more real world scenario, we might keep the players around and throw up some UI to let
25  // users activate controllers and pick their players again before resuming.
26  //
27  // This example could easily be extended to use bindings. The process would be very similar,
28  // just creating a new instance of your action set subclass per player and assigning the
